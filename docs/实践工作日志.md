## 阶段二：数据生成模块开发

### 日期：2026-06-28

#### 2.1 三表数据模型设计

**What**：设计 `category.csv`（8 分类）、`product.csv`（24,000 SKU）、`price.csv`（~430 万行）三张表。

**Why**：
- **三表而非单表**：电商数据天然有维度-事实分层——分类和商品是缓慢变化维度，价格是日度快照事实。三表分离既模拟真实数仓星型模型，也让后续 JOIN 清洗成为必要步骤，体现大数据处理的完整性。
- **8 分类 × 3000 商品**：8 是课设模拟数据量的合理值（太少看不出分类聚合效果，太多超出 OSS 免费额度）。3000 商品/分类 × 181 天 ≈ 430 万行价格记录，CSV 文件约 100MB 满足任务书要求。
- **随机种子固定为 42**：保证团队协作和 CI/CD 中每次生成的数据完全一致，使测试结果可复现。

**How**：使用 `pandas` 构建 DataFrame → `to_csv()` 直接写入。价格表因数据量大（400 万+行），采用逐商品生成并追加写入的流式策略而非全量内存拼接，控制峰值内存在 500MB 以内。

**可复用经验**：模拟数据生成时要刻意注入已知比例的脏数据（本项目 ~6%），这样清洗模块的过滤率可精确验证，避免"不知道洗掉了多少、对不对"的问题。

## 阶段三：OSS 存储模块开发

### 日期：2026-06-28

#### 3.1 OSS 上传实现

**What**：实现 `oss_upload.py`，将本地 `raw_data/` 下三个 CSV 上传至阿里云 OSS Bucket 的 `raw/` 路径。

**Why**：选择 `oss2` 官方 SDK 而非 `boto3` 或 CLI 工具——阿里云 ClickHouse 的 OSS 集成原生对接 oss2 接口规范，且 `oss2` 的 `put_object_from_file` 对小文件（< 100MB）更简洁。

**How**：`oss2.Auth(AK, SK)` → `oss2.Bucket(auth, endpoint, bucket)` → `bucket.put_object_from_file(oss_path, local_path)`。

#### 3.2 环境变量加载路径问题 [关键问题 #1]

**What**：首次运行 `oss_upload.py` 时报 `AttributeError: 'NoneType' object has no attribute 'strip'`。

**Why**：
- **直接原因**：`os.getenv("OSS_ACCESS_KEY_ID")` 返回 `None`，因为 `load_dotenv()` 默认从当前工作目录（CWD）查找 `.env`，而用户从项目根目录运行脚本时，Python 进程的 CWD 可能与脚本所在目录不一致。
- **根因分析**：`python src/oss_operation/oss_upload.py` 的 CWD 是项目根目录，但 `load_dotenv()` 在脚本内部没有指定绝对路径，依赖隐式的 CWD 行为。当 IDE 终端或 VS Code 的 CWD 变化时，这种行为就会失败。
- **设计反思**：任何依赖外部配置文件的模块，都不应假定 CWD。应基于 `__file__` 推导项目根目录，再拼接配置文件绝对路径。

**How**：
```python
PROJECT_ROOT = os.path.abspath(os.path.join(os.path.dirname(__file__), "..", ".."))
ENV_PATH = os.path.join(PROJECT_ROOT, ".env")
load_dotenv(ENV_PATH)
```
同时在调用凭据前增加显式校验：
```python
if not AK or not SK:
    print("❌ 环境变量未配置，请编辑 .env 后重试")
    sys.exit(1)
```

**可复用经验**：
- Python 项目中所有 `load_dotenv()` 应传入基于 `__file__` 的绝对路径，这个模式可写入项目模板的 `.env.example` 注释中。
- 外部依赖（数据库连接、云 SDK）的初始化参数应在调用前显式校验，给出中文提示而非让框架抛 `NoneType` 异常，这在团队协作中减少沟通成本。

## 阶段四：ClickHouse 计算引擎开发

### 日期：2026-06-28

#### 4.1 初始方案：ClickHouse OSS 外部表引擎

**What**：最初设计使用 ClickHouse 的 `ENGINE = OSS(...)` 直接读取 OSS Bucket 中的 CSV，通过 `CREATE EXTERNAL TABLE` 建立云端数据映射，避免数据下载到本地中转。

**Why（初始选型理由）**：
- 理想情况下，OSS 外部表能实现"零拷贝"——ClickHouse 直接从 OSS 拉取数据，无需 Python 中转，减少网络跳数和本地磁盘占用。
- 阿里云 ClickHouse 文档明确列出了 `OSS` 表引擎，理论上应支持 `ENGINE = OSS('url', 'ak', 'sk', 'CSVWithNames')` 语法。

**How（初始实现）**：
```sql
CREATE TABLE oss_raw_category
ENGINE = OSS('https://{bucket}.{endpoint}/raw/category.csv', '{ak}', '{sk}', 'CSVWithNames')
(category_id Int32, category_name String);
```

#### 4.2 OSS 引擎语法错误 [关键问题 #2]

**What**：执行时报 `Syntax error: failed at position 261 ('Int32')`，ClickHouse 将列定义的 `Int32` 识别为非法 token。

**Why**：
- **版本差异**：阿里云 ClickHouse 23.8.16.1 的 `OSS` 表引擎语法与开源 ClickHouse 最新版不兼容。开源版中列定义放在 `ENGINE = OSS(...)` 之后的括号中是合法语法，但阿里云 23.8 版本的解析器对此有不同的语法期望（可能需要列定义放在 `ENGINE` 之前，或完全不同的参数格式）。
- **文档滞后**：阿里云 ClickHouse 文档中 OSS 引擎的示例可能与实际版本不完全对应。托管服务的版本通常落后于开源最新版，且会引入专有修改。
- **决策失误**：在未验证目标环境 ClickHouse 版本和 OSS 引擎兼容性之前，就假设开源文档语法可用。应该先 `SELECT version()` 确认版本，再用最小 DDL 做语法探测。

**How（排查过程）**：
1. 从错误信息定位位置 `position 261`，确认是第二条 `CREATE TABLE` 的 `Int32` 被识别为非法
2. 怀疑多语句问题，将 3 条 DDL 拆分为独立 `client.command()` 调用——问题依旧
3. 确认问题在 `ENGINE = OSS(...)` 语法本身，而非多语句
4. 检索阿里云 ClickHouse 23.8 文档，确认该版本对 `OSS` 引擎的支持有限

**最终方案**：放弃 ClickHouse OSS 引擎，改用 Python `oss2` SDK 下载 CSV → pandas 内存处理 → `client.insert()` 逐批写入。

#### 4.3 多语句执行问题 [关键问题 #3]

**What**：将 3 条 `CREATE TABLE` 用分号拼成一个字符串传给 `client.command()`，报语法错误。

**Why**：ClickHouse HTTP 接口的 `/` 端点（`client.command()` 底层调用）不支持批量 SQL。分号在 ClickHouse HTTP 协议中不被识别为语句分隔符——它是 ClickHouse 原生客户端（`clickhouse-client`）的特性，而非 HTTP 接口的。

**How**：将每一条 DDL/DML 拆分为独立的 `client.command()` 调用。对于 3 张 OSS 外部表 + 3 张本地表 + 多条 INSERT，使用循环或逐一调用：
```python
for table_name, ddl in tables.items():
    client.command(ddl)
```

**可复用经验**：
- `clickhouse_connect` 的 `command()` 方法等价于 HTTP `/` 端点，不支持多语句；如果确实需要批量执行，使用 `client.raw_query()` 或通过 TCP 协议（`clickhouse-driver`）。
- 在不确定 SDK 能力边界时，先看源码中 HTTP 请求的实际 payload 构造方式，比反复试错更高效。

#### 4.4 架构重构：从 OSS 引擎到 Python 中转 [关键决策]

**What**：将 `ck_price_calc.py` 从 ClickHouse OSS 引擎方案彻底重构为 Python `oss2` + pandas + `client.insert()` 方案。

**Why（全方位复盘）**：

| 维度 | OSS 引擎方案 | Python 中转方案 |
|------|-------------|----------------|
| 兼容性 | 依赖 CK 版本和阿里云定制语法 | 只要有 `oss2` + `clickhouse_connect` 即可 |
| 清洗灵活性 | SQL WHERE 子句，复杂逻辑需嵌套 | pandas 任意复杂过滤 + 统计分析 |
| 调试体验 | 黑盒，SQL 报错信息有限 | 可逐步 debug DataFrame 每一步 |
| 网络效率 | CK 直连 OSS，一次网络跳转 | OSS → Python → CK，两次跳转 |
| 内存占用 | CK 服务端管理 | Python 端需控制 DF 大小 |

- **核心权衡**：兼容性 > 网络效率。100MB 数据两次网络跳转的额外耗时约 5-10 秒，但一个语法兼容性问题可能阻塞数小时。在课设场景下开发效率远比运行效率重要。
- **数据清洗前置到 pandas**：原来清洗逻辑通过 `INSERT ... WHERE price IS NOT NULL AND price > 0 AND price < 5000`，重构后在 pandas 中执行，每步过滤输出统计（空值数、负值数、极端值数），方便验证清洗规则是否与 `data_generator.py` 注入的 ~6% 脏数据率对齐。
- **模块职责重新划分**：CK 不再负责数据导入，只做它最擅长的事——列式存储 + OLAP 聚合（`CREATE TABLE ... AS SELECT ... GROUP BY`）。

**How（新架构数据流）**：
```
OSS Bucket                    Python                         ClickHouse
┌──────────┐  oss2.get()    ┌──────────────┐  insert()      ┌──────────────┐
│ category │ ──────────────→│ pandas        │ ──────────────→│ dim_category │
│ product  │               │ .read_csv()   │                │ dim_product  │
│ price    │               │ .merge()清洗   │                │ fact_price_  │
└──────────┘               │ .values.tolist│                │ clean        │
                            └──────────────┘                │ daily_index  │
                                                            └──────────────┘
```

**可复用经验**：
- **托管服务 ≠ 开源版本**：阿里云 CK、AWS Redshift、GCP BigQuery 等托管服务对标准 SQL/DDL 的支持都有差异。技术选型时先跑最小可行性验证（如 `SELECT version()` + 一个最小建表），不要假设"官方文档写的语法一定能在目标版本跑通"。
- **数据清洗逻辑要可观测**：每一步过滤后输出计数，这样当脏数据比例与预期不符时能快速定位是哪条规则的问题。本项目严格对齐了 `data_generator.py` 的 6% 脏数据注入率，清洗后刚好 ~94% 保留。
- **Python 中转不是反模式**：在大数据教材中常强调"计算下推"（pushdown），但在原型开发和课设场景下，Python 端处理提供了极大的调试灵活性。真正生产环境再考虑 SQL pushdown 或 Spark 分布式处理。

## 阶段六：CI/CD 流水线配置

### 日期：2026-06-28

#### 6.1 四阶段流水线设计

**What**：配置 GitHub Actions 工作流，实现 `数据校验 → 单元测试 → 集成测试 → 自动部署` 四阶段自动化。

**Why**：
- **阶段分离**：每个阶段通过 `needs` 依赖链串行，前一阶段失败则后续不执行—避免在语法错误代码上浪费 CI 分钟数。
- **ClickHouse 服务容器**：使用 GitHub Actions 的 `services.clickhouse` 声明式启动容器，自动管理生命周期和端口映射，比手动 `docker run` 更可靠。
- **Artifacts 传递**：阶段间通过 `upload-artifact` / `download-artifact` 传递产物（校验报告 → 测试数据 → 计算结果 → 图表），避免重复生成。
- **部署仅在 main 分支触发**：`if: github.ref == 'refs/heads/main'` 确保只有合并到主干才跑全量 100MB 数据生成和 OSS 上传，PR 只跑数据校验和单元测试。

**How**：每个 job 独立 `checkout` + `setup-python` + `pip install`；OSS/CK 凭据通过 GitHub Secrets 注入环境变量；失败时使用 `if: always()` 确保报告始终归档。

#### 6.2 CI 环境与本地环境的差异处理

**What**：CI 中 ClickHouse 版本（Docker `latest`，通常为 24.x）与阿里云托管版（23.8.16.1）不一致。

**Why**：CI 用开源最新版检验 SQL 逻辑正确性（单元测试），生产环境是阿里云托管版。两者语法不完全兼容（如本次遇到的 OSS 引擎问题）。

**How（处理策略）**：
- 单元测试不依赖 OSS 引擎——测试的是清洗规则和加权公式的数学正确性，这些与 CK 版本无关。
- 集成测试阶段用 Python 中转方案，规避了 OSS 引擎兼容性问题。
- CI 中的 CK 仅用于验证 `CREATE TABLE`、`INSERT`、`GROUP BY` 等标准 SQL 可执行，不做引擎特性测试。

**可复用经验**：CI 环境应测试业务逻辑正确性，不应测试特定云厂商的专有特性。专有特性的兼容性验证应在预发布环境（staging）中完成，而非 CI 阶段。

## 阶段七：测试模块开发

### 日期：2026-06-28

#### 7.1 测试策略与分层

**What**：编写 18 个测试用例，分为单元测试（逻辑验证）和集成测试（连通性验证）两层。

**Why**：
- **`pytest.mark.integration` 标记**：区分两种测试——无外部依赖的逻辑测试随时可跑（`pytest -k "not integration"`），需要真实凭据的连通性测试按需执行。这在 CI 和本地开发中都很实用——CI 中 OSS 凭据通过 Secrets 注入但本地可能未配置。
- **测试数据独立构造**：测试不依赖 `data_generator.py` 的输出。每个测试函数内部用硬编码的小数据集（5-10 行）覆盖边界条件。这样测试运行快（< 1s）且不因数据文件缺失而 skip。
- **清洗阈值对齐**：测试中的 `PRICE_UPPER_BOUND = 5000` 必须与 `ck_price_calc.py` 中的 `price < 5000` 同步——这是典型的"配置散落"问题，在重构时统一由注释标注。

**How**：
```python
# test_ck_sql.py - 加权公式验证
def test_weighted_avg_price_formula():
    # SUM(price * sales_volume) / SUM(sales_volume)
    weighted_avg = (10*5 + 20*10) / (5+10)  # 250/15 = 16.67
    assert weighted_avg == pytest.approx(16.666666, rel=1e-4)
```

**可复用经验**：测试用例用硬编码的小数据集覆盖公式正确性——不依赖任何文件 I/O 或网络调用。这是单元测试的基本原则：**测试的是逻辑，不是环境**。

## 整体技术决策复盘

### 决策 1：Python 中转 vs ClickHouse 原生引擎

| 方案 | 优势 | 劣势 | 最终选择 |
|------|------|------|----------|
| CK OSS 引擎 | 零拷贝、架构简洁 | 版本兼容性差、调试黑盒 | ✗ |
| Python 中转 | 兼容性好、可观测 | 多一次网络跳转 | ✓ |

**结论**：在非生产环境中，**可维护性 > 理论最优架构**。150MB 数据的中转开销可忽略，但一个兼容性问题可能阻塞数小时。

### 决策 2：pandas 内存清洗 vs SQL WHERE 清洗

| 方案 | 优势 | 劣势 | 最终选择 |
|------|------|------|----------|
| SQL WHERE | 计算下推 | 无法输出中间统计 | ✗ |
| pandas 逐层过滤 | 每步可观测 | 内存占用更大 | ✓ |

**结论**：数据清洗的**可观测性**比计算效率更重要。知道每步过滤了多少条数据，才能验证清洗规则是否正确。

### 决策 3：多语句 command vs 逐条 command

| 方案 | 优势 | 劣势 | 最终选择 |
|------|------|------|----------|
| 分号拼接多条 | 代码更短 | HTTP 协议不支持 | ✗ |
| 逐条 command | 可靠、错误精准 | 代码行数增加 | ✓ |

**结论**：不要假设 SQL 分隔符在所有协议上都有效。**先确认传输协议的能力边界**，再决定代码写法。

### 决策 4：CWD 相对路径 vs `__file__` 绝对路径

| 方案 | 优势 | 劣势 | 最终选择 |
|------|------|------|----------|
| 相对路径 | 简洁 | CWD 不可控 | ✗ |
| `__file__` + join | 位置确定 | 稍多代码 | ✓ |

**结论**：任何需要读写文件的模块，基路径都应从 `__file__` 推导。这是 Python 工程化的基本纪律。

## 整体心得

本次课程设计最大的收获不是"学会了 ClickHouse"或"会调 OSS SDK"，而是在真实约束下做工程决策的能力：

1. **"能用"和"可靠"是两回事**：第一个版本的所有代码都能在理想条件下运行，但 CWD 变一下就崩、CK 版本差一点就语法错误、凭据没配就 `NoneType`。工程化就是在这些边界条件下增加防御。

2. **托管服务不是开源软件的简单部署**：阿里云 ClickHouse 有专有修改，文档滞后，语法不兼容——这些都不是个例，是所有云厂商的常态。以后再遇到类似场景，第一件事是做**最小可行性验证**（一个 `SELECT 1` + 一个最小 DDL + 一个最小 INSERT）。

3. **可观测性比优雅架构更重要**：每步清洗输出计数、每个外部调用加 try/except 并给出中文提示——这些"啰嗦"的代码在调试时节省的时间远超编写它们的时间。

4. **CI/CD 的价值不在配置本身**：而在于它迫使你把所有依赖显式化——凭据放 Secrets、数据用 Artifacts 传递、环境差异用条件分支处理。这个过程本身就是在做系统设计。

5. **技术选型的核心是"约束"**：课设的约束是时间和评分项，不是 QPS 和延迟。在约束下做最务实的选择，比追求"最优架构"更有意义。
