# 电商价格指数大数据分析系统 — 实践工作日志

> **课程设计大作业** | 权重：20%
>
> 本项目构建了从模拟数据生成 → 阿里云 OSS 存储 → ClickHouse 清洗计算 → 可视化呈现的完整大数据流水线。以下记录了真实的踩坑过程、每个关键决策的 Why 复盘，以及哪些经验可以在以后的项目中直接复用。

---

## 一、流水帐：完整的开发过程

### 1.1 启动阶段：技术选型的困境

拿到《业务需求说明书》后，第一反应是找 AI 给出一套完整的大数据处理系统架构。AI 初始生成了一套非常"高级"的方案——Kafka 做数据接入、Flink 做流计算、Hadoop HDFS 做底层存储、Spark 做批处理。全是本地化部署的开源组件，数据不需要上传到阿里云。

坦诚说，AI 给的东西看不太懂。每个组件名字都认识，但拼在一起不知道从哪下手。又去 CSDN 上搜了两篇文章——一篇讲"ClickHouse + MinIO 构建本地数据湖"，一篇讲"3 分钟快速安装 ClickHouse"。事实是根本无法速成。光 ClickHouse 的安装配置就卡了半天，更别说 MinIO、Kafka 这些不需要但看起来很重要的东西。

对于"数据从哪里来"的理解也出了问题。AI 给的方案建议从中国金融信息网爬真实 CPI 数据，但爬虫需要处理反爬、数据清洗、格式转换，而且爬到的数据结构和课设要求的"电商三表"不匹配。

最后在同学的提醒下，才确定了真正可行的路线：**阿里云 OSS + 云端 ClickHouse + Python 脚本**。四个步骤清晰明确：生成模拟数据 → 上传 OSS → ClickHouse 清洗计算 → 可视化出图。放弃了本地部署的幻想，选择了全云端托管方案。

### 1.2 开发阶段：ClickHouse 模块的连续踩坑

数据生成模块（`data_generator.py`）和 OSS 上传模块（`oss_upload.py`）相对顺利，真正的挑战出现在 ClickHouse 数据清洗与价格指数计算模块。

**第一个坑：多语句执行失败**。把 3 个 `CREATE TABLE` 用分号拼在一起传给 `client.command()`，想一次执行完。报语法错误——HTTP 协议不支持分号分隔多条 DDL。改成逐条执行，每张表单独一个 `client.command()`。

**第二个坑：OSS 引擎语法不兼容**。逐条执行后依然报错。排查发现不是多语句的问题，是 `ENGINE = OSS(...)` 语法本身在阿里云 ClickHouse 23.8 版本上不兼容。列定义的写法不被识别。这个问题没法绕过去——要么升级 ClickHouse 版本（但这是阿里云托管版，自己不能升级），要么换方案。最终彻底放弃了 OSS 引擎方案，改为 Python 用 `oss2` SDK 从 OSS 下载 CSV 到 pandas，再通过 `client.insert()` 写入 ClickHouse。

**第三个坑：CTAS 语法不兼容**。原计划用 `CREATE TABLE ... AS SELECT ... GROUP BY` 一句搞定建表和计算。阿里云 CK 23.8 在 CTAS 中对 GROUP BY 的列解析有 bug，报 `Column not under aggregate function`。只能拆成两步：先 `CREATE TABLE` 显式定义列，再 `INSERT INTO ... SELECT ... GROUP BY`。

**第四个坑：date 类型转换**。pandas 的 `read_csv()` 把 `price_date` 读成了字符串，但 ClickHouse 的 `Date` 列期望 `datetime.date` 对象。写入时报 `TypeError: str - datetime.date`。插入前加了一行 `pd.to_datetime().dt.date` 转换解决。

**第五个坑：CTAS → CREATE + INSERT 拆分后仍报错**。拆分后的 `INSERT INTO ... SELECT ... GROUP BY` 在 CK 23.8 中列名映射仍出现问题，最终在 CREATE TABLE 时显式列出全部列名和类型，并在 INSERT 中不使用表别名来避免解析歧义。

### 1.3 网络问题：ClickHouse 连接超时

白名单机制是个隐藏的坑。之前还能正常连接，隔天突然超时。排查发现是公网 IP 变了——家庭宽带的 IP 是动态分配的，24 小时左右就会换一次。而且更隐蔽的原因是**开了梯子**（VPN/代理），导致 IP 频繁变化，ClickHouse 的安全组每次都要重新加白名单。

解决方案：去阿里云 ClickHouse 控制台 → 安全管理 → 白名单 → 把 `curl ifconfig.me` 查到的当前公网 IP 加进去。或者直接在百度搜"IP"也能看到。注意白名单不是即时生效的，有 1-2 分钟延迟，改完后等一会儿再测。

### 1.4 可视化问题：价格 vs 价格指数

最初画出来的图，纵轴是 `weighted_avg_price`（绝对值，大约 1480~1560 元）。但课设要求的是"价格指数"，不是"绝对价格"。价格指数的核心是**相对涨跌**，行业规范是以基期 = 100 为基准，这样才能直观反映物价变动幅度——和统计局 CPI 的表述逻辑一致。

修复方案：在 ClickHouse SQL 中增加基期归一化计算——用子查询取出 2026-01-01（基期）每个分类的加权均价作为分母，当日加权均价除以基期均价再乘以 100。`draw_cpi_trend.py` 改为绘制 `price_index` 列而非 `weighted_avg_price`。图表标题改为"基期 2026-01-01 = 100"，底部加了脚注说明算法。

### 1.5 Git 推送问题

GitHub 在国内的访问极不稳定。先是 HTTPS 推送被 GFW 阻断（`Connection was reset`），换成 SSH 后能推了但还是间歇性阻断。时而能连时而不能连，全靠运气。还遇到 GitHub Push Protection 检测到 `config/.env.example` 里写了真实 AccessKey，被服务端拒绝推送——赶紧把模板文件改成占位符 `your_access_key_id_here`，又去阿里云后台把泄露的那对密钥禁用了。

## 二、复盘：关键设计决策的 Why 分析

按照 2W1H 框架，以下对开发过程中的 5 个关键决策进行复盘。每个决策从"当时的具体约束是什么、考虑过哪些替代方案、为什么排除、约束变了会不会改、哪些经验可以复用"五个维度展开。

---

### 决策 1：为什么放弃本地部署方案，选择阿里云全托管？

**What**：在技术选型阶段，放弃了 AI 推荐的 Kafka + Flink + HDFS 本地化部署方案，改为阿里云 OSS + 云端 ClickHouse + Python 脚本的全托管方案。

**Why**：
- **当时的具体约束**：这门课设的开发周期是 2 天（非 8 周），团队只有我自己，笔记本是 Windows 16GB 内存。不是"2C4G 服务器预算 200 元/月"那种标准约束，而是"一个人要在 48 小时内从零到交付所有代码 + 文档"的时间约束。
- **考虑过的替代方案**：
  - **方案 A（AI 推荐的本地化部署）**：Kafka + Flink + HDFS + Spark + ClickHouse——需要在本机安装至少 3 个服务，且每个都有自己的配置文件和端口管理。在 CSDN 上搜了两篇文章想速成，但光 ClickHouse 的安装配置就折腾了半天，后面还有 MinIO、Kafka，全部搞完至少要 2 周。
  - **方案 B（同学建议的阿里云全托管）**：OSS 和 ClickHouse 都是阿里云现成的，不需要自己部署维护。Python 脚本只需要会 `oss2` 和 `clickhouse_connect` 两个库即可。
  - **方案 C（中国金融信息网爬虫）**：AI 建议用爬虫获取真实 CPI 数据——但爬虫需要额外学 Scrapy/Selenium，而且爬到的数据结构不一定匹配课设要求的电商三表（商品、价格、分类）。结构不匹配意味着需要重做数据清洗逻辑。
- **排除 A 和 C 的原因**：方案 A 需要掌握 Kafka、Flink、HDFS、Spark 至少 4 个新组件，每个的学习曲线都在 1 周以上，超出 48 小时的约束。方案 C 的数据来源与课设要求的电商场景不一致，且爬虫的数据质量不可控（反爬、字段缺失、格式不统一），会导致后续所有模块都要重新适配。方案 B 只用学 2 个 Python 库，且阿里云控制台有图形界面，不需要记命令行参数，一个下午就可以走通全流程。
- **如果约束变了，选择会变吗**：如果是团队 3 人、8 周时间，我会选方案 A 的部分组件——保留 ClickHouse（列式 OLAP 确实是正确的选型），去掉 Kafka 和 Flink（课设的数据是静态 100MB CSV，不需要流处理），用 MinIO 替代阿里云 OSS（省去云服务费用）。
- **可复用的经验**：**技术选型的第一步不是"什么最好"，而是"我能在死线之前学会什么"。** AI 给出的方案往往是理论上最优的，但最优方案的学习成本可能超出时间预算。以后接到新任务，第一件事是列出所有候选方案中每个组件"我需要从零学什么"，然后砍掉需要学超过 3 个新东西的方案。这条经验适用于任何时间紧迫的项目，不限于课设。

### 决策 2：OSS 引擎 vs Python 中转——数据导入的架构重构

**What**：在 ClickHouse 数据清洗模块中，从最初设计的 `ENGINE = OSS(...)` 直接读取 OSS 数据，重构为 Python `oss2` SDK 下载 CSV → pandas 内存处理 → `client.insert()` 写入 CK。

**Why**：
- **当时的具体约束**：阿里云 ClickHouse 的版本是 23.8.16.1（托管版，用户不能自己升级）。该版本的 `ENGINE = OSS(...)` 语法与开源 ClickHouse 24.x 不兼容——列定义的写法不被识别，直接报语法错误。这是一个**硬阻断**，不是"不够好"的问题，是"根本跑不通"。
- **考虑过的替代方案**：
  - **方案 A（CK OSS 引擎）**：CK 直读 OSS，数据不经过 Python。零拷贝，架构简洁。已实测不可用——`Syntax error: failed at position 261`。
  - **方案 B（Python oss2 下载 + client.insert()）**：Python 做中转，多一次网络跳转（OSS → Python → CK，100MB 数据额外耗时约 8 秒），但完全绕过 CK 版本兼容问题。
  - **方案 C（阿里云 DataWorks 数据同步）**：阿里云提供的 ETL 服务，可从 OSS 同步到 CK——但需要额外开通 DataWorks 服务，且配置流程预计需要 1-2 小时学习。
- **排除 A 和 C 的原因**：方案 A 已被实测证明不可用——CK 23.8 的 OSS 引擎语法不接受列定义写在 `ENGINE = OSS(...)` 之后。方案 C 需要额外开通付费服务且学习成本高——课设的预算约束不是金钱（阿里云有免费额度），而是时间。多学一个 DataWorks 配置流程，意味着少写一份文档的时间。
- **如果约束变了，选择会变吗**：如果使用的是开源 ClickHouse 最新版（24.x+），我会选方案 A（OSS 引擎）。因为 OSS 引擎在开源 24.x 中语法兼容且零拷贝效率更高。如果数据量从 100MB 扩大到 10GB+，方案 B 的内存瓶颈（pandas DataFrame 无法承载 10GB）会迫使切换回方案 A 或引入 Spark。但在当前 100MB 约束下，方案 B 完全够用。
- **可复用的经验**：**当托管云服务告诉你"语法错误"时，不要浪费时间怀疑是不是自己写错了——大概率是版本不兼容。** 任何云数据库在使用前，先跑一句 `SELECT version()` 确认版本，再跑一个单列 `CREATE TABLE` 做最小可行性验证。这个习惯可以省去数小时的无效排查。另外，ETL 管道中加一个 Python 中转层在原型阶段提供了极大的调试灵活性（每步清洗输出计数、断点查看中间结果），这个"牺牲一点效率换可观测性"的权衡在中小数据量场景下普遍适用。

### 决策 3：SQL 清洗 vs pandas 清洗——数据处理逻辑放在哪里

**What**：将脏数据清洗逻辑从 SQL `WHERE` 子句改为 pandas 内存过滤，清洗完成后通过 `client.insert()` 写入 ClickHouse。

**Why**：
- **当时的具体约束**：数据生成时按 ~6% 比例注入了三类脏数据（2% 空值、2% 负价格、2% 极端值 999999）。清洗模块需要验证"实际洗掉了多少"，并且这个数字必须和生成端对齐——如果洗掉了 10% 或 3%，说明清洗规则有问题。但 SQL `WHERE` 是一个黑盒——只能看到最后剩多少，不知道每类脏数据各自被洗掉了多少。
- **考虑过的替代方案**：
  - **方案 A（纯 SQL WHERE）**：数据保留在 ClickHouse 中，用 `INSERT INTO ... SELECT ... WHERE ...` 过滤——代码简洁，但无法输出中间统计。要知道空值洗掉多少、负值洗掉多少，需要额外写 3 条 `SELECT count()`。
  - **方案 B（pandas 内存过滤）**：数据下载到 pandas，逐层过滤并输出计数——每步可观测（"空值: 86,467，负值: 86,461，极端值: 86,951 → 合计 ~6%，对上了！"），但多一次 Python ↔ CK 数据传输。
- **排除 A 的原因**：方案 A 虽然更"SQL 原生"，但在原型验证阶段无法快速确认清洗规则是否正确。我已经遇到过多语句、OSS 引擎、CTAS 三个 CK 兼容性问题，对"SQL 能不能按预期执行"的信任已经很低。pandas 方案让每一步都可视化，出问题能立刻定位到具体哪条过滤规则。
- **如果约束变了，选择会变吗**：如果数据量从 100MB（434 万行）扩大到 10GB+，pandas 内存过滤不可行（DataFrame 会 OOM），会切回 SQL WHERE 方案。但会在 SQL 中嵌入手动计数子查询来保持可观测性。如果 ClickHouse 版本升级到支持 CTAS 无误的版本，也会更多把逻辑下推到 CK。
- **可复用的经验**：**数据处理管道中，每一步的过滤规则都应该能输出"过滤前后行数对比"。** 这个习惯让调试从"猜哪里出了问题"变成"看哪一步的数字不对"。不论用 SQL 还是 pandas 还是 Spark，这个原则都适用。对于脏数据清洗尤其重要——如果不知道注入比例，就无法验证清洗规则的正确性。

### 决策 4：绝对价格 vs 价格指数——可视化的语义修正

**What**：将可视化图表从绘制 `weighted_avg_price`（绝对价格，约 1480~1560 元）改为绘制 `price_index`（基期归一化指数，基期 2026-01-01 = 100）。

**Why**：
- **当时的具体约束**：课设要求的是"价格指数"，不是"价格均值"。经济学中价格指数必须以基期 = 100 为基准，表示相对涨跌幅度。如果用绝对价格绘图，8 个分类的价格绝对值差异太大（生鲜 ~50 元、数码 ~2000 元），放在一张图里 y 轴尺度完全不可比。而且绝对价格的变动幅度（如数码从 1500 涨到 1520）在图表上肉眼几乎看不出来，没有分析价值。
- **考虑过的替代方案**：
  - **方案 A（直接画 `weighted_avg_price` 绝对值）**：省事，但不符合"价格指数"的语义定义。而且 8 条线的 y 轴尺度差异太大，生鲜那条线和数码那条线之间空了一大片空白，图表利用率极低。
  - **方案 B（按分类分别画 8 张小图）**：每张图一个分类，各自有自己的 y 轴——代价是 8 张图对比不便，而且每张图仍然是"绝对值"而非"指数"。
  - **方案 C（基期归一化）**：在 ClickHouse SQL 中用子查询取出基期（2026-01-01）每个分类的加权均价，当日均价除以基期均价再乘 100。8 条线都从 100 起步，涨跌幅度一目了然。
- **排除 A 和 B 的原因**：方案 A 不符合课设要求——绝对值不是指数。方案 B 虽然解决了 y 轴尺度问题，但仍然不是指数。方案 C 在语义和展示效果上都是正确的。
- **如果约束变了，选择会变吗**：如果课设只要求"数据可视化"而不限定"价格指数"，方案 A 作为快速探索性分析是可以接受的。但在任何需要"指数"概念的场景（CPI、股票指数、汇率），基期归一化是必选项，没有替代。
- **可复用的经验**：**可视化的 y 轴到底代表什么，必须在写第一行绘图代码前想清楚。** "平均值"、"指数"、"增长率"在图表上的含义完全不同。一个简单的检查方法是：给一个非技术背景的人看图，问他"这个值是 150 是什么意思？"——如果回答不上来，说明 y 轴的语义没有对齐业务定义。

### 决策 5：`load_dotenv()` 的路径问题——配置管理的小细节大影响

**What**：将 `load_dotenv()` 从无参调用改为基于 `__file__` 推导绝对路径的调用方式，并在凭据变量使用前增加显式 `None` 校验。

**Why**：
- **当时的具体约束**：`.env` 文件存放在项目根目录，但 `oss_upload.py` 和 `ck_price_calc.py` 都在 `src/` 子目录下。Windows 终端和 VS Code 集成终端的默认 CWD 行为不一致——有时在项目根目录，有时在 `src/` 下。
- **考虑过的替代方案**：
  - **方案 A（依赖 CWD，让用户确保在根目录执行）**：代码最简单，但把环境问题转嫁给了用户。实测 VS Code 的 Python 运行按钮默认 CWD 是脚本所在目录（`src/`），必然加载不到 `.env`。而且报错信息是 `'NoneType' object has no attribute 'strip'`——一个刚打开项目的同学看到这个错误，至少要花 10 分钟才能定位到"是 `.env` 没加载到"。
  - **方案 B（`load_dotenv("../.env")` 相对路径）**：依赖 CWD，换一个 CWD 就找不到。
  - **方案 C（`__file__` + `os.path.join` 推导绝对路径 + 显式校验）**：当前采用。
- **排除 A 和 B 的原因**：A 不可靠——CWD 是 Python 进程的全局状态，不由脚本控制。B 和 A 本质相同，只是换了一种相对路径写法。方案 C 是唯一确定性的方案——不论从哪里启动脚本，`__file__` 总是指向脚本文件自身的位置。
- **如果约束变了，选择会变吗**：如果是 Docker 部署，环境变量通过 Docker Compose 的 `env_file` 或 K8s ConfigMap 注入，`load_dotenv()` 根本不需要——直接用 `os.getenv()` 读容器环境变量。这时应该在代码开头加一个判断：如果 `os.getenv("CK_HOST")` 已有值，跳过 `load_dotenv()`。
- **可复用的经验**：**Python 项目中所有路径推导都应从 `__file__` 出发，永远不要依赖 CWD。** 这个原则可以写入项目脚手架模板，每次新建 Python 项目时自动包含 `PROJECT_ROOT = os.path.abspath(os.path.join(os.path.dirname(__file__), ".."))`。同时，任何读外部凭据的代码，在传给 SDK 之前必须显式检查是否为 `None` 并给出中文错误提示——这 3 行代码能省去团队每个人 10 分钟的排查时间。

## 三、未来可复用的经验清单

| # | 经验 | 适用场景 | 来源 |

## 五、整体心得

这次课设最深的感受不是"学会了 ClickHouse"这种技术细节，而是几个更底层的道理：

1. **"看不懂的方案就是不可行的方案"**。AI 给出的 Kafka + Flink + HDFS 方案，每个组件名字都认识但不知道怎么配、不知道配置完能不能连起来。在自己能独立跑通的范围内做选择，比追求"听起来很厉害"重要得多。

2. **踩坑不是能力问题，是工程常态**。ClickHouse 模块 5 个坑，每个坑解决后回头看都觉得"这有什么难的"，但在报错的那一刻，从定位原因到验证修复，每个都需要从错误堆栈反推、查文档、做最小复现、测试。这个过程没有一个是可以被 AI 跳过的——因为坑的根因往往是"阿里云 CK 23.8 和开源 24.x 的语法差异"这种只有实际运行才能发现的问题。

3. **细节决定可靠性**。`load_dotenv()` 加不加 `__file__`、凭据传给 SDK 前检不检查 `None`、白名单的 IP 是不是被代理换了——这些都不是什么高大上的架构决策，但每个都能让一个本来可以跑通的流程崩掉。工程化就是把这些细节一个一个堵上。

4. **课设是一次难得的"踩坑练习"**。在真实工作环境中，有些坑（如 OSS 引擎版本不兼容、IP 白名单变化）可能需要跟运维沟通、提工单等回复。课设环境里自己既是开发又是运维，踩坑的速度更快、印象也更深刻。这些坑踩过一次，以后在团队中遇到类似问题时可以更快定位。
