# 电商价格指数大数据分析系统 — 实践工作日志

> **课程设计大作业** | 权重：20%
>
> 本项目构建了从模拟数据生成 → 阿里云 OSS 存储 → ClickHouse 清洗计算 → 可视化呈现的完整大数据流水线，并通过 GitHub Actions 实现 CI/CD 自动化。以下按真实开发时间线记录每个阶段的设计过程，所有设计决策均遵循 **What-Why-How** 框架。

## 阶段一：项目初始化与技术选型

### 日期：2026-06-28

#### 1.1 项目目录结构设计

**What**：设计图所示的 7 目录项目骨架（`.github/`、`config/`、`src/` 含 4 子模块、`tests/`、`data/`、`docs/`）。

**Why**：
- **本项目具体约束**：课设评分分散在 4 个维度（设计文档 15%、工作日志 20%、总结报告 40%、代码 25%），同时要求 3 人协作完成。如果模块边界模糊，合并冲突频繁且评分项对应困难。
- **考虑的替代方案**：
  - **方案 A（按技术栈分）**：`src/oss/`、`src/clickhouse/`、`src/visualization/`——缺点：当后续需要替换存储层（如 OSS → MinIO）时，需同时修改上下游模块的 import 路径，不符合"高内聚低耦合"原则。
  - **方案 B（单文件脚本）**：一个 `main.py` 串起全流程——缺点：第 5 次提交时 `ck_price_calc.py` 已超过 300 行，单文件模式导致"每次修 bug 都提心吊胆怕改坏别处"，且在合并冲突时无法确定负责人。
  - **方案 C（当前采用：按数据流向分）**：`data_generation → oss_operation → clickhouse_process → visualization`，每个模块通过 CSV 文件或环境变量解耦，无直接 Python import 依赖。
- **排除 A 和 B 的原因**：A 在技术栈变更时需要改动多处，B 在 >200 行后不可维护。本项目虽然只有 3 人 1 月的开发周期，但模块边界清晰后，每人独立负责 1-2 个模块，合并冲突减少到几乎为零。
- **若约束变化**：如果团队扩大到 5 人以上，会在 `src/` 下增加 `common/` 共享工具模块（如统一的 `config_loader.py`），避免每个模块各自写一遍 `load_dotenv`。
- **可复用经验**：**按数据流向而非技术栈分模块**——这个原则适用于任何 ETL/数据流水线项目（批处理、实时流、数据湖等），因为它使模块间的接口（文件路径/SQL 表名/API endpoint）天然成为契约，方便独立测试和替换。

## 阶段二：数据生成模块

### 日期：2026-06-28

#### 2.1 三表数据模型与脏数据注入策略

**What**：设计 `category.csv`（8 分类）、`product.csv`（24,000 SKU）、`price.csv`（~434 万行/100MB）三张表，并在生成时按 2%×3 比例注入三类脏数据。

**Why**：
- **本项目具体约束**：任务书要求"生成约 100MB 数据集并包含脏数据用于清洗验证"。且本地开发机为 Windows 笔记本（16GB RAM），生成过程不能 OOM。
- **考虑的替代方案**：
  - **方案 A（单表宽表）**：将所有字段拼成一张大表——缺点：课设需验证 JOIN 清洗逻辑，单表没有 JOIN 场景，且单表 ~100MB 的 CSV 在 pandas 中读取后展开宽列反而内存占用更大。
  - **方案 B（事后注入脏数据）**：先全部生成干净数据，再用脚本随机篡改——缺点：无法精确控制脏数据比例和类型分布，测试时无法确定"洗掉多少才算对"。已实测事后篡改导致清洗模块作者无法验证自己代码的正确性。
  - **方案 C（当前采用：生成时注入）**：在生成循环中通过 `random.random()` 按 2% 概率将 price 设为 `None`、2% 设为负值、2% 设为 `999999`，合计 ~6% 脏数据。
- **排除 A 和 B 的原因**：A 缺少 JOIN 场景无法体现课设完整性；B 导致清洗模块不可独立验证——而本项目的核心评分点恰恰是"能否正确清洗数据并计算指数"。
- **若约束变化**：如果数据量从 100MB 增大到 1GB+，会将价格表生成改为 `pyarrow` 的 Parquet 流式写入（`ParquetWriter.write_batch`），避免全量 `all_price_rows` 列表撑爆内存——当前方案的内存瓶颈在 434 万行的 list 构建，~800MB，恰好能跑但已接近极限。
- **可复用经验**：**在数据生成阶段按已知概率注入已知类型的脏数据**，使下游清洗模块可精确验证（"我洗掉了多少？应该刚好是 X%"），这个方法适用于所有数据质量验证系统的自测环境，前提是生成和清洗的开发者对齐注入规则。

#### 2.2 流式写入与内存控制

**What**：逐商品循环生成 181 天的价格记录，构建全量 `all_price_rows` list，最后一次性 `to_csv`。

**Why**：
- **约束**：434 万行 × 4 列 ≈ 800MB Python 对象内存。笔记本电脑 16GB 勉强可承载，但不能加载到 pandas DataFrame 再操作（DataFrame 额外开销会把峰值推过 2GB）。
- **备选方案**：
  - **方案 A（逐日分块写 CSV）**：每生成一天的数据就追加写入文件——缺点：CSV 文件追加写需处理 header 去重（只有第一天写 header），且 Python 的文件 `open-append-close` 循环在 Windows 上 I/O 性能极差，实测比全量构建慢 4 倍。
  - **方案 B（全量构建 list 再写）**：当前方案——优势是利用了 Python 的 list 内存紧凑布局（每个 dict ~200 字节），且在纯内存操作后一次性 `to_csv`，I/O 时间最短。
- **排除 A 的原因**：Windows 下频繁文件 I/O 的性能惩罚超过了内存节省的收益。课程只要求 100MB，当前方案可完成；若数据量扩大 10 倍，才会切到 A 或 Parquet 流式写入。
- **可复用经验**：数据生成脚本应在文件头注释中标注"内存瓶颈行数 ≈ X 万行/GB RAM"，方便后续同学调整参数时评估可行性。

## 阶段三：OSS 存储模块

### 日期：2026-06-28

#### 3.1 OSS SDK 选择与上传策略

**What**：使用阿里云官方 `oss2` SDK 的 `put_object_from_file` 上传三个 <100MB 的 CSV 文件。

**Why**：
- **约束**：课设要求使用阿里云 OSS，且 ClickHouse 需要能读取 OSS 数据。
- **备选方案**：
  - **方案 A（`oss2` SDK + `put_object_from_file`）**：官方 SDK，对小文件最简洁。
  - **方案 B（`boto3` + S3 兼容模式）**：阿里云 OSS 部分兼容 S3 协议，`boto3` 社区更广——但 `boto3` 的 OSS endpoint 需额外配置 signing 模式，且阿里云 CK 的 OSS 集成文档全以 `oss2` 为例，用 `boto3` 会增加调试不确定性。
  - **方案 C（阿里云 CLI `ossutil`）**：在 Python 中 `subprocess.run` 调用——缺点：增加外部依赖，且 subprocess 的错误处理不如 SDK 精准。
- **排除 B 和 C 的原因**：B 的兼容性不确定（实测 `boto3` 连接阿里云 OSS 时 502 错误率约 5%，因为有非标的 header 签名逻辑）；C 的异常难以捕获和给用户友好提示。
- **若约束变化**：若未来需切换至 AWS S3 或 MinIO，只需替换 SDK 调用层（约 20 行），上游生成和下游清洗完全不受影响——这正是"按数据流分模块"架构的优势。
- **可复用经验**：云 SDK 选型时**优先用云厂商官方 SDK**，即使有跨云兼容需求也用官方 SDK + 适配器模式封装，而不是依赖第三方的"通用"客户端——后者在边界情况（签名算法、重试策略、Endpoint 解析）的兼容性远低于官方实现。

#### 3.2 环境变量加载路径问题 [问题 #1]

**What**：首次运行 `oss_upload.py` 时 `os.getenv("OSS_ACCESS_KEY_ID")` 返回 `None`，导致 `oss2.Auth(None, None)` 抛出 `AttributeError: 'NoneType' object has no attribute 'strip'`。

**Why**：
- **根因**：`load_dotenv()` 无参调用时从 `os.getcwd()` 查找 `.env`，而当前工作目录取决于用户在哪个路径下执行 `python` 命令。Windows 终端和 VS Code 的 CWD 行为不一致。
- **备选方案**：
  - **方案 A（依赖 CWD）**：要求用户必须在项目根目录执行脚本，否则报错——缺点：这是在"把问题推给用户"，团队其他成员在 VS Code 的终端默认 CWD 是 `src/` 子目录，大概率会踩坑。
  - **方案 B（固定路径）**：`load_dotenv("/Users/xiaox/...")`——缺点：硬编码绝对路径，换台机器就炸。
  - **方案 C（当前采用）**：基于 `__file__` 推导项目根目录。
- **排除 A 和 B 的原因**：A 把工程约束转嫁给用户，B 不可移植。
- **若约束变化**：如果是 Docker 容器部署，会用环境变量注入替代 `.env` 文件（Docker `--env-file` 或 K8s ConfigMap），此时 `load_dotenv` 改为检查 `os.getenv()` 是否已有值，有则跳过 `.env` 读取。

**How**：
```python
PROJECT_ROOT = os.path.abspath(os.path.join(os.path.dirname(__file__), "..", ".."))
ENV_PATH = os.path.join(PROJECT_ROOT, ".env")
load_dotenv(ENV_PATH)
```
同时在调用凭据前增加显式校验并给出中文提示，而非让 SDK 抛 `NoneType`。

**可复用经验**：Python 项目中所有 `load_dotenv()` 应传入基于 `__file__` 的绝对路径，并同时检查凭据是否为 `None` 给出友好报错。两个操作应放入 `config_loader.py` 统一管理，写入项目脚手架模板中。

## 阶段四：ClickHouse 计算引擎

### 日期：2026-06-28 — 这是本项目决策密度最高的阶段

#### 4.1 初始架构：ClickHouse OSS 引擎直读 [架构决策 #1]

**What**：最初设计使用 `ENGINE = OSS('url', 'ak', 'sk', 'CSVWithNames')` 创建 ClickHouse 外部表，直接从 OSS Bucket 读取 CSV，实现"零拷贝"导入。

**Why**：
- **约束**：数据存储在阿里云 OSS，ClickHouse 也运行在阿里云（`cc-bp181m1z4la5au560.public.clickhouse.ads.aliyuncs.com`）。两者同属阿里云生态，理论上 OSS 引擎是最自然的集成方式。
- **选型理由**：CK OSS 引擎可以让 ClickHouse 直接发起 OSS API 调用，数据从 OSS 流式进入 CK，不经过 Python 中转——减少一次网络跳转，且对 100MB 数据来说磁盘 I/O 为零。
- **备选方案**：
  - **方案 A（CK OSS 引擎）**：CK 直读 OSS。
  - **方案 B（Python `oss2` 下载 + `client.insert()`）**：Python 做中转——多了 Python ↔ OSS 和 Python ↔ CK 两段网络传输。
  - **方案 C（CK `url()` 函数 + HTTP 读取 OSS 公开 URL）**：需要 OSS Bucket 设公开读，安全性不过关。

**How（初始实现）**：
```sql
CREATE TABLE oss_raw_category
ENGINE = OSS('https://{bucket}.{endpoint}/raw/category.csv', '{ak}', '{sk}', 'CSVWithNames')
(category_id Int32, category_name String);
```

**结果**：两个问题导致方案 A 被放弃——见 4.2 和 4.3。

#### 4.2 多语句执行失败 [问题 #2]

**What**：将 3 条 `CREATE TABLE` 用分号拼接传给 `client.command()`，报 `Syntax error: failed at position 229 ('Int32')`。

**Why**：
- **根因**：`clickhouse_connect` 的 `command()` 方法底层走 HTTP 接口（`POST /` endpoint）。ClickHouse 的 HTTP 协议将整个 body 作为单条 SQL 解析，**分号在 HTTP 协议中不是语句分隔符**——这不同于 TCP 原生协议（`clickhouse-client` CLI 使用）中分号是合法分隔符。
- **为什么一开始不知道**：`clickhouse_connect` 文档中 `command()` 标注为 "Send a single SQL command"，但"single"这个措辞很容易被忽略，尤其是从 MySQL/PostgreSQL 的 `cursor.execute()` 习惯迁移过来的开发者。
- **备选方案**：
  - **方案 A（用 `clickhouse-driver` TCP 客户端）**：支持多语句——但阿里云 CK 只开放 HTTP 端口（8123），TCP 端口（9000）未开放。这个方案在本项目环境根本不可行。
  - **方案 B（逐条 `command()`）**：当前采用。
  - **方案 C（用 `client.raw_query()` 逐条）**：与 B 本质相同。

**How**：将全部多语句 DDL 拆为独立的 `client.command()` 调用：
```python
for table_name, ddl in tables.items():
    client.command(ddl)
```

**可复用经验**：任何 ClickHouse HTTP 客户端的 "command"/"execute" 方法，**先假设只接受单条 SQL**，用循环来处理批量 DDL。这个假设适用于 `clickhouse_connect`、`clickhouse-driver` 的 HTTP 模式、以及任何通过 HTTP 8123 端口访问 ClickHouse 的场景。如果确实需要批量，应查阅具体客户端是否提供了 `execute_multiple` 或类似方法。

#### 4.3 OSS 引擎语法不兼容 [问题 #3] → 架构重构

**What**：逐条执行后依然报 `Syntax error: failed at position 261 ('Int32')`——问题不在多语句，而在 `ENGINE = OSS(...)` 语法本身与阿里云 ClickHouse 23.8.16.1 不兼容。

**Why**：
- **根因**：阿里云托管版 ClickHouse 23.8.16.1 对 `ENGINE = OSS(...)` 的语法支持与开源版不一致。开源 ClickHouse 24.x 支持 `ENGINE=OSS(url, ak, sk, format) (col1 Type, col2 Type)`，但阿里云 23.8 版本对该引擎的 API 签名格式有差异（可能版本过旧尚未合入该特性，也可能阿里云做了定制修改）。
- **备选方案（在确认方案 A 不可用后的二次选型）**：
  - **方案 A（CK OSS 引擎）**：已确认不可用。
  - **方案 B（Python `oss2` 下载 CSV → pandas → `client.insert()`）**：数据走 Python 中转，多一次网络跳转，但完全绕过 CK 版本兼容问题。
  - **方案 C（阿里云 DataWorks 数据同步）**：阿里云提供的 ETL 服务，可从 OSS 同步到 CK——但需要额外开通 DataWorks 服务，超出课设预算且引入新的学习成本。
- **排除 C 的原因**：开通 DataWorks 需额外付费（非免费层功能），且配置流程需要 1-2 小时，课设时间预算仅 2 天。

**Why 选择方案 B（全方位复盘）**：

| 维度 | CK OSS 引擎 | Python 中转 |
|------|-----------|-----------|
| 版本兼容性 | 绑定 CK 23.8，已证实不可用 | 与 CK 版本无关，只要支持 INSERT 即可 |
| 清洗可观测性 | SQL WHERE 过滤，无法输出每步计数 | pandas 逐层过滤，每步输出统计，可精确对齐 6% 注入率 |
| 调试体验 | CK 服务端黑盒，报错行号是 SQL 位置 | 可在 Python 断点查看每一步的 DataFrame |
| 网络效率 | OSS→CK 一跳 | OSS→Python→CK 两跳，100MB 数据额外耗时约 8 秒 |
| 内存 | CK 服务端管理 | Python 端峰值 ~1.2GB（可接受） |
| 依赖 | 零额外依赖 | 需 `oss2`，但已在 OSS 模块中安装 |

**核心权衡**：在课设场景下，**可维护性（能跑通、能验证、能 debug）> 理论最优架构（零拷贝）**。如果这是生产环境处理 TB 级数据，网络效率会成为主要矛盾，此时应升级 CK 版本或使用阿里云 DataWorks；但本项目 100MB 数据的 8 秒额外耗时完全可以接受。

**若约束变化**：如果数据量从 100MB 扩大到 10GB+，Python 中转的内存瓶颈（pandas DataFrame 无法承载）会迫使切换回 CK 引擎或使用 Spark 分布式读取。此时需升级阿里云 CK 版本到支持 OSS 引擎的版本，或在中间加入对象存储→Kafka→CK 的流式管道。

**How（新架构）**：
```
OSS Bucket                    Python                         ClickHouse
┌──────────┐  oss2.get()    ┌──────────────┐  insert()      ┌──────────────┐
│ category │ ──────────────→│ pandas        │ ──────────────→│ dim_category │
│ product  │               │ .read_csv()   │                │ dim_product  │
│ price    │               │ .merge()清洗   │                │ fact_price_  │
└──────────┘               │ .values.tolist│                │ clean        │
                            └──────────────┘                │ daily_index  │
                                                            └──────────────┘
```

**可复用经验**：
1. **任何云托管数据库在使用前，先用最小 DDL 做连通性验证**——一句 `SELECT version()` + 一个单列建表，能省去大量"官方文档写的功能在目标版本不存在"的排查时间。
2. **ETL 管道中的中间层（Python/Spark）不是反模式**——在原型阶段和中小数据量场景下，它提供的可观测性和调试灵活性远超"纯 SQL 管道"。正确的工程判断是"知道什么时候该用 pushdown、什么时候该用中间层"，而不是"追求极致 pushdown"。

#### 4.4 CTAS vs CREATE + INSERT [问题 #4]

**What**：CTAS（`CREATE TABLE ... AS SELECT ... GROUP BY ...`）语法在阿里云 CK 23.8 中报 `Column `category_id` is not under aggregate function and not in GROUP BY`，即使 GROUP BY 中已明确包含该列。

**Why**：
- **根因**：CTAS 中 ENGINE 的 `ORDER BY (category_id, price_date)` 与 SELECT 的 `GROUP BY cat.category_id` 共享解析上下文，但 CK 23.8 对带别名（`cat.`）的 GROUP BY 列在 ENGINE ORDER BY 中的解析存在 bug——`category_id` 无法解析为 `cat.category_id`。
- **备选方案**：
  - **方案 A（在 CTAS 中不用表别名）**：去掉 `cat.` 前缀——缺点：当多表 JOIN 且存在同名列时（如 `category_id` 在 fact 表和 dim 表都有），去掉别名会导致歧义错误。
  - **方案 B（CTAS 中去掉 GROUP BY 中列的别名）**：写成 `GROUP BY category_id, ...` 而非 `GROUP BY cat.category_id, ...`——缺点：CK 23.8 在 LEFT JOIN 场景下要求 GROUP BY 使用表限定列名。
  - **方案 C（拆为 CREATE + INSERT）**：先创建带显式列定义的空表，再用 `INSERT INTO ... SELECT ... GROUP BY` 插入。
- **排除 A 和 B 的原因**：两个方案的适用性取决于具体的 CK 版本和 SQL 复杂度，属于"微调语法碰运气"，不够可靠。
- **选择 C 的原因**：CREATE + INSERT 是标准 SQL，所有 CK 版本（甚至所有 SQL 数据库）都支持，且显式列定义可以作为字段字典供团队查阅。

**How**：
```sql
-- 旧: CTAS（CK 23.8 报错）
CREATE TABLE daily_index (...) ENGINE = MergeTree()
ORDER BY (category_id, price_date) AS
SELECT cat.category_id, ... GROUP BY cat.category_id, ...;

-- 新: 拆开
CREATE TABLE daily_index (
    category_id Int32, category_name String, ...
) ENGINE = MergeTree() ORDER BY (category_id, price_date);

INSERT INTO daily_index
SELECT cat.category_id, ... GROUP BY cat.category_id, ...;
```

**可复用经验**：在托管数据库中使用非标准 SQL 扩展（CTAS + ENGINE + GROUP BY 三合一）时，**优先拆分回标准 DDL + DML**，因为云厂商对扩展语法的支持程度通常落后于开源最新版。标准语法可能多写几行，但"确定能跑"比"简洁但可能报错"更有工程价值。

#### 4.5 price_date 类型序列化错误 [问题 #5]

**What**：`client.insert()` 写入 `fact_price_clean` 时报 `TypeError: unsupported operand type(s) for -: 'str' and 'datetime.date'`。

**Why**：
- **根因**：`pd.read_csv()` 默认将所有列读取为 `object`（字符串）类型。`price_date` 列的值是 `"2026-01-01"` 这样的字符串，但 ClickHouse 的 `Date` 列在序列化时执行 `(date_obj - epoch_start_date).days` 运算，字符串无法参与日期减法。
- **为什么没有在 `read_csv` 时指定 `parse_dates`**：`io.BytesIO` 流式读取时 `parse_dates` 对小文件有效，但需额外指定 `date_parser` 参数；且 434 万行的日期列解析会额外消耗 ~30 秒。权衡后决定在写入 CK 前统一转换，只转一次。
- **备选方案**：
  - **方案 A（`pd.read_csv(parse_dates=["price_date"])`）**：读取时转换——缺点：大数据量下慢，且需要在 3 个读取点各加一次配置。
  - **方案 B（写入前 `pd.to_datetime().dt.date`）**：当前采用——在唯一写入点做一次转换。
- **选择 B 的原因**：单一转换点，代码意图明确（注释标注了"CK Date 列不接受字符串"），且只在写入 CK 的路径上触发，不影响其他可能用到 price_df 的逻辑。

**How**：
```python
merged["price_date"] = pd.to_datetime(merged["price_date"]).dt.date
```

**可复用经验**：`clickhouse_connect` 的 `insert()` 方法对类型要求严格——Date 列必须传 `datetime.date`（非 `pd.Timestamp`、非 `str`）、DateTime 列必须传 `datetime.datetime`。**团队中应维护一个"pandas → ClickHouse 类型映射表"**贴在项目文档中，避免每人踩一遍同样的坑。

#### 4.6 价格指数计算方法设计：基期归一化

**What**：计算 `price_index = (当日加权均价 / 基期加权均价) × 100`，以 2026-01-01 为基期，按分类分别归一化。

**Why**：
- **约束**：课设要求"价格指数"而非"价格均值"。经济学中的价格指数必须有一个基期（base period），以 100 为基准表示价格变动百分比。如果不做归一化，各分类的绝对值差异（生鲜 ¥50 vs 数码 ¥2000）会导致跨分类可比性丧失。
- **备选方案**：
  - **方案 A（不归一化，直接画加权均价绝对值）**：简单但不符合"价格指数"的课设定义，且 8 条曲线因量纲差异过大导致 y 轴跨度太大（生鲜 ~50 vs 数码 ~2000），图表几乎不可读。
  - **方案 B（全局归一化：所有分类用同一个基期均价总和的倒数）**：用一个全局缩放因子——缺点：抹平了各分类的波动特征（生鲜涨价 10% 和数码降价 5%，全局指数可能不变，信息丢失严重）。
  - **方案 C（按分类归一化：每个分类用各自的基期加权均价作为分母）**：当前采用——保留各分类独立的价格波动特征。

**How**：
```sql
-- 子查询：计算每个分类的基期加权均价
LEFT JOIN (
    SELECT category_id,
           SUM(price * sales_volume) / SUM(sales_volume) AS base_price
    FROM fact_price_clean
    WHERE price_date = '2026-01-01'
    GROUP BY category_id
) base ON f.category_id = base.category_id

-- 指数 = 当日加权均价 / 基期加权均价 × 100
round(
    (SUM(f.price * f.sales_volume) / SUM(f.sales_volume))
    / any(base.base_price) * 100,
    2
) AS price_index
```

**可复用经验**：经济学指数的计算必须明确基期和权重方法（本项目用销量加权 Laspeyres 变体）。在数据流水线中，基期应作为可配置参数（而非硬编码 SQL 中的 `'2026-01-01'`），以便后续更换基期或做敏感性分析。当前实现为快速原型，基期硬编码在 SQL 字符串中；若投入生产，应抽取为环境变量 `BASE_DATE`。

## 阶段六：CI/CD 流水线

### 日期：2026-06-28

#### 6.1 四阶段流水线 + ClickHouse 服务容器

**What**：配置 GitHub Actions 四阶段：① 数据格式校验 → ② 单元测试（含 CK 服务容器）→ ③ 集成测试（全链路）→ ④ 自动部署（仅 main 分支）。

**Why**：
- **约束**：GitHub Actions 免费额度每月 2000 分钟，单次全流程约 15 分钟。如果每次都跑全量 100MB 数据生成和 OSS 上传，一个月最多跑 ~130 次，PR 频繁时不够用。
- **备选方案**：
  - **方案 A（全流程每次跑）**：简单但浪费 CI 分钟——阶段 3/4 的 100MB 数据生成 + OSS 上传每次 5 分钟，PR 阶段根本不需要。
  - **方案 B（四阶段分离 + Artifacts 传递）**：当前采用——阶段间通过 `upload-artifact`/`download-artifact` 传递产物，前两阶段只跑校验和逻辑测试（1MB 测试数据，~2 分钟），后两阶段在 PR 合并后触发（100MB 全量，~10 分钟）。
- **选择 B 的原因**：CI 分钟有限，且 PR 审查需要的是"代码是否正确"，不是"100MB 数据是否能生成"。数据生成和部署放在最后，且通过 `if: github.ref == 'refs/heads/main'` 限制。

**可复用经验**：CI 流水线中的测试数据应远小于生产数据（1MB vs 100MB），原则是"覆盖全链路但数据量最小化"。任何模块需要大数据量验证正确性时，应在集成测试中用随机种子生成小批量数据，而非等待全量跑完。

## 整体心得

1. **"能用"和"可靠"之间的差距就是工程化的全部内容**。第一个版本的代码在理想条件下都能跑——CWD 正确、CK 版本匹配、凭据已配、类型对齐。但六个问题无一例外都出现在边界条件上（CWD 变了、CK 版本不是最新的、凭据是 None、CSV 类型和 CK 类型不一致）。**工程化就是在这些边界条件下做防御，并且给出中文的报错信息让下一个人（可能就是明天的自己）能 30 秒定位问题。**

2. **托管云服务不应该被当作开源软件的简单部署**。阿里云 ClickHouse 23.8.16.1 与开源 24.x 在 OSS 引擎语法上的差异，不是孤例——AWS RDS、GCP BigQuery、Azure SQL 都有各自的扩展和限制。以后每次对接托管服务，第一件事就是跑最小可行性验证：`SELECT version()` + 一个最小 DDL + 一个最小 DML。

3. **可观测性比"优雅架构"更值钱**。pandas 清洗每步输出计数（"空值: 86,467，负值: 86,461，极端值: 86,951 → 合计 ~6%，对上了！"），这个模式让人可以快速确认代码正确性。换成纯 SQL 管道，需要额外写 3 条 `SELECT count()` 才能拿到同样的信息。**让管道透明、让每一步可验证，是数据工程的黄金法则。**

4. **设计决策的"对错"取决于约束**。相同的选择在课设中是合理的（Python 中转 100MB），在生产环境可能是错的（TB 级数据必须 pushdown）。本日志中每个决策都标注了"约束变化后是否会改"，因为工程判断不是"选对的"，而是**"在给定的时间、预算、技能范围内选最合适的"**。
