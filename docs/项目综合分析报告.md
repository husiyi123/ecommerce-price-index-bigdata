## （一）需求分析

### 1.1 业务背景

在电商运营中，价格是反映市场供需关系的核心信号。传统 CPI（居民消费价格指数）由国家统计部门按月发布，粒度粗、滞后性强，难以满足电商实时运营决策的需要。本项目构建一个高频（日度）、分类别的电商价格指数监测系统，模拟从数据生成到可视化的全链路大数据处理流程。

### 1.2 功能需求

| 编号 | 需求名称 | 详细描述 |
|------|---------|---------|
| F1 | 模拟数据生成 | 生成 8 个电商分类、24,000 个 SKU、181 天（2026-01-01 ~ 2026-06-30）共 434 万条价格记录，总数据量约 100MB，含约 6% 脏数据 |
| F2 | 云存储上传 | 将生成的 CSV 文件上传至阿里云 OSS 对象存储，支持上传后 ETag 完整性校验 |
| F3 | 数据格式校验 | 对生成的 CSV 进行六步校验：文件存在性、Schema 字段完整性、类型推断与转换、非空约束、外键引用完整性、价格业务逻辑 |
| F4 | 数据清洗与入库 | 从 OSS 下载数据，过滤脏数据（空值、负价格、极端异常值 ≥5000），写入 ClickHouse |
| F5 | 价格指数计算 | 按分类 × 日期维度，计算销量加权平均价格指数：以 2026-01-01 为基期，`price_index = (当日加权均价 / 基期加权均价) × 100` |
| F6 | 趋势可视化 | 绘制 8 个分类的日度加权均价折线图，支持中文标签、月度刻度、高分辨率输出 |
| F7 | CI/CD 自动化 | GitHub Actions 四阶段流水线：数据校验 → 单元测试 → 集成测试 → 自动部署 |

### 1.3 非功能需求

| 编号 | 类型 | 要求 |
|------|------|------|
| NF1 | 数据规模 | 支持 100MB 级别原始数据的生成与处理 |
| NF2 | 数据质量 | 清洗过程可观测，每步输出剔除统计，脏数据率对齐生成时的 6% |
| NF3 | 可扩展性 | 价格指数公式（Laspeyres/Paasche/Fisher）预留扩展接口 |
| NF4 | 可移植性 | 凭据通过 `.env` 管理，不硬编码；模块基于 `__file__` 推导项目根目录，不依赖 CWD |
| NF5 | 自动化 | 全流程可通过 GitHub Actions 一键执行，从数据生成到可视化输出全自动 |
| NF6 | 可视化效果 | 300 DPI 高分辨率中文图表，图例区分 8 个分类 |

### 1.4 数据流概要


### 2.2 关键技术决策

#### 决策 1：Python pandas 中转 vs ClickHouse OSS 外部表引擎

| 方案 | 优点 | 缺点 |
|------|------|------|
| **OSS 表引擎（S3/OSS 函数）** | 零拷贝，ClickHouse 直接读 OSS 文件 | 阿里云 CK 23.8 版本兼容性问题，DDL 语法多变，调试成本高 |
| **Python oss2 + pandas 中转** ✅ | 稳定可靠，每步可观测，不受 CK 版本限制 | 多一次网络传输，内存占用增加 |

**最终选择**：Python 中转方案。实践中 OSS 外部表引擎因阿里云 ClickHouse 23.8 版本的函数签名不兼容被放弃。

#### 决策 2：ClickHouse HTTP 多语句限制

ClickHouse 的 `client.command()` 底层使用 HTTP 协议（默认 8123 端口），**不支持分号分隔的多条 SQL 语句**。每一条 DDL/DML 必须单独调用 `client.command()`。这影响了建表和插入代码的结构设计。

#### 决策 3：销量加权均价 vs Laspeyres/Paasche/Fisher 指数

| 指数类型 | 公式 | 适用场景 | 本项目 |
|---------|------|---------|--------|
| **销量加权均价** | `Σ(p×v) / Σ(v)` | 单期截面比较 | ✅ 已实现 |
| Laspeyres | `Σ(p_t × q_0) / Σ(p_0 × q_0)` | 固定基期数量权重 | 预留扩展 |
| Paasche | `Σ(p_t × q_t) / Σ(p_0 × q_t)` | 固定报告期数量权重 | 预留扩展 |
| Fisher | `√(Laspeyres × Paasche)` | 理想指数，取几何平均 | 预留扩展 |

**选择理由**：销量加权均价计算简单直观，适合日度高频场景；更复杂的指数公式已写入设计文档，可通过配置切换。

#### 决策 4：基于 `__file__` 的路径推导

所有模块使用 `os.path.abspath(os.path.join(os.path.dirname(__file__), "..", ".."))` 推导项目根目录，而非依赖 `os.getcwd()`。这确保了从任意目录执行脚本（如 `python src/clickhouse_process/ck_price_calc.py`）时路径正确。

### 3.2 物理表设计

#### 3.2.1 源数据 CSV Schema

**category.csv** — 分类维度表（8 行）

| 列名 | 类型 | 说明 |
|------|------|------|
| `category_id` | Int32 | 分类主键，1~8 |
| `category_name` | String | 分类名称（生鲜果蔬、粮油副食、家居百货、数码电器、服装鞋帽、美妆护肤、母婴用品、零食饮料） |

**product.csv** — 商品维度表（24,000 行）

| 列名 | 类型 | 说明 |
|------|------|------|
| `product_id` | Int64 | 商品主键，自增 |
| `category_id` | Int32 | 外键，引用 `category.category_id` |
| `product_name` | String | 商品名称（格式：`商品_{分类ID}_{商品ID}`） |
| `brand` | String | 品牌（A牌/B牌/C牌/D牌/自营） |
| `spec` | String | 规格（1kg/500ml/2件装/大号/标准版） |

**price.csv** — 价格事实表（~4,344,000 行）

| 列名 | 类型 | 说明 |
|------|------|------|
| `product_id` | Int64 | 外键，引用 `product.product_id` |
| `price_date` | Date | 价格日期（2026-01-01 ~ 2026-06-30） |
| `price` | Float64 | 价格（含 ~6% 脏数据：2% 空值、2% 负值、2% 极大值 999999） |
| `sales_volume` | Int64 | 日销量（0-1200 随机） |

#### 3.2.2 ClickHouse 物理表

**dim_category** — 分类维度表
```sql
CREATE TABLE IF NOT EXISTS dim_category (
    category_id Int32,
    category_name String
) ENGINE = MergeTree()
ORDER BY category_id;
```

| 设计要点 | 说明 |
|---------|------|
| 引擎 | MergeTree — ClickHouse 核心表引擎，支持分区、排序、主键 |
| 排序键 | `category_id` — 主键等值查询和 JOIN 优化 |
| 分区 | 无 — 8 行的小表不需要分区 |

**dim_product** — 商品维度表
```sql
CREATE TABLE IF NOT EXISTS dim_product (
    product_id Int64,
    category_id Int32,
    product_name String,
    brand String,
    spec String
) ENGINE = MergeTree()
ORDER BY product_id;
```

| 设计要点 | 说明 |
|---------|------|
| 引擎 | MergeTree |
| 排序键 | `product_id` — 主键查询和与价格表的 JOIN 优化 |
| 分区 | 无 — 24,000 行维度表，全量扫描可接受 |

**fact_price_clean** — 清洗后价格事实表
```sql
CREATE TABLE IF NOT EXISTS fact_price_clean (
    product_id Int64,
    category_id Int32,
    price_date Date,
    price Float64,
    sales_volume Int64
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(price_date)
ORDER BY (category_id, price_date);
```

| 设计要点 | 说明 |
|---------|------|
| 引擎 | MergeTree |
| 排序键 | `(category_id, price_date)` — 核心查询模式是按分类 + 日期聚合 |
| 分区键 | `toYYYYMM(price_date)` — 按月分区（202601~202606），支持分区裁剪和 TTL 管理 |
| 冗余 `category_id` | 反范式设计：清洗阶段通过 JOIN product 表预先带入 `category_id`，避免每次聚合时重复 JOIN |

**daily_category_price_index** — 日度分类价格指数结果表
```sql
CREATE TABLE daily_category_price_index (
    category_id Int32,
    category_name String,
    price_date Date,
    weighted_avg_price Float64,
    total_sales Int64,
    price_index Float64
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(price_date)
ORDER BY (category_id, price_date);
```

| 设计要点 | 说明 |
|---------|------|
| 引擎 | MergeTree |
| 排序键 | `(category_id, price_date)` — 支持按分类的时间范围查询 |
| 分区键 | `toYYYYMM(price_date)` — 按月分区 |
| 物化粒度 | 每个 (category_id, price_date) 一行，即 8 分类 × 181 天 = 1,448 行 |
| price_index | 价格指数列，以 2026-01-01 为基期（=100），公式：`(当日加权均价 / 基期加权均价) × 100` |

### 3.3 表设计原则总结

1. **星型模型**：`dim_category` → `dim_product` → `fact_price_clean` 构成维度建模的星型结构
2. **反范式冗余**：`fact_price_clean` 冗余 `category_id`，避免聚合时多表 JOIN
3. **分区裁剪**：事实表按月分区，查询特定月份时减少扫描量
4. **排序键对齐查询**：排序键匹配 `GROUP BY category_id, price_date` 模式
5. **分层处理**：`fact_price_clean`（ODS 清洗层）→ `daily_category_price_index`（DWS 汇总层）

### 4.3 数据清洗与计算流程

```
price.csv (4,344,000 行)
    │
    ├─ 空值过滤: price IS NOT NULL           → 剔除 ~87,000 行 (2%)
    ├─ 非正数过滤: price > 0                 → 剔除 ~87,000 行 (2%)
    ├─ 极端值过滤: price < 5000              → 剔除 ~87,000 行 (2%)
    │
    ▼
清洗后 (~4,083,000 行, ~94%)
    │
    ├─ LEFT JOIN product ON product_id       → 获取 category_id
    ├─ 剔除 orphan (category_id IS NULL)
    │
    ▼
写入 fact_price_clean
    │
    ├─ GROUP BY category_id, category_name, price_date
    ├─ 计算加权均价: SUM(price × sales_volume) / SUM(sales_volume)
    ├─ 计算基期价格指数: (当日加权均价 / 基期 2026-01-01 加权均价) × 100
    │
    ▼
daily_category_price_index (1,448 行)
    │
    ├─ 导出 price_index_result.csv（含 weighted_avg_price + price_index）
    ├─ Matplotlib 绑图（以 price_index 为 Y 轴）
    │
    ▼
cpi_price_trend.png
```

## （六）核心 SQL 代码逻辑

### 6.1 DDL：三张核心表

```sql
-- 分类维度表（8 行小表）
CREATE TABLE IF NOT EXISTS dim_category (
    category_id Int32,
    category_name String
) ENGINE = MergeTree()
ORDER BY category_id;

-- 商品维度表（24,000 行）
CREATE TABLE IF NOT EXISTS dim_product (
    product_id Int64,
    category_id Int32,
    product_name String,
    brand String,
    spec String
) ENGINE = MergeTree()
ORDER BY product_id;

-- 价格事实表（~408 万行，按月分区）
CREATE TABLE IF NOT EXISTS fact_price_clean (
    product_id Int64,
    category_id Int32,           -- 冗余字段，避免聚合时 JOIN
    price_date Date,
    price Float64,
    sales_volume Int64
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(price_date)   -- 按月分区：202601~202606
ORDER BY (category_id, price_date); -- 对齐 GROUP BY 查询模式
```

### 6.2 DML：核心聚合计算（含基期价格指数）

```sql
-- 创建结果表（v1.1 新增 price_index 字段）
DROP TABLE IF EXISTS daily_category_price_index;
CREATE TABLE daily_category_price_index (
    category_id Int32,
    category_name String,
    price_date Date,
    weighted_avg_price Float64,
    total_sales Int64,
    price_index Float64                                -- 新增加
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(price_date)
ORDER BY (category_id, price_date);

-- 核心聚合 INSERT（含基期价格指数计算）
INSERT INTO daily_category_price_index
SELECT
    cat.category_id,
    cat.category_name,
    f.price_date,
    SUM(f.price * f.sales_volume) / SUM(f.sales_volume) AS weighted_avg_price,
    SUM(f.sales_volume) AS total_sales,
    round(
        (SUM(f.price * f.sales_volume) / SUM(f.sales_volume))
        / any(base.base_price) * 100,
        2
    ) AS price_index
FROM fact_price_clean f
LEFT JOIN dim_category cat ON f.category_id = cat.category_id
LEFT JOIN (
    -- 子查询：计算每个分类的基期（2026-01-01）加权均价
    SELECT
        category_id,
        SUM(price * sales_volume) / SUM(sales_volume) AS base_price
    FROM fact_price_clean
    WHERE price_date = '2026-01-01'
    GROUP BY category_id
) base ON f.category_id = base.category_id
GROUP BY cat.category_id, cat.category_name, f.price_date;
```

### 6.3 SQL 逻辑详解

#### 6.3.1 加权平均公式

```
weighted_avg_price = Σ(price × sales_volume) / Σ(sales_volume)
```

**示例**：某分类在某天有两条价格记录：
- 商品 A：price=10, sales=5 → 贡献 = 50
- 商品 B：price=20, sales=10 → 贡献 = 200
- 加权均价 = (50+200)/(5+10) = 250/15 = **16.67**

对比简单平均 = (10+20)/2 = 15.00，可见加权均价更能反映销量权重。

#### 6.3.2 数据清洗规则（在 Python 端执行）

```sql
-- 等价的 SQL WHERE 子句（清洗逻辑在 Python pandas 端实现，此处为逻辑等价表达式）
-- price IS NOT NULL AND price > 0 AND price < 5000
```

| 条件 | 含义 | 剔除量（估算） |
|------|------|--------------|
| `price IS NOT NULL` | 排除空价格 | ~2%（~87,000 行） |
| `price > 0` | 排除负数和零 | ~2%（~87,000 行） |
| `price < 5000` | 排除异常极大值（生成时注入 999999） | ~2%（~87,000 行） |

#### 6.3.3 GROUP BY 维度

```sql
GROUP BY cat.category_id, cat.category_name, f.price_date
```

- `category_id` + `category_name`：同一分类的两种标识，必须同时出现在 GROUP BY 中（或在大多数 SQL 方言中用功能依赖）
- `price_date`：日度粒度，181 天
- 聚合结果：8 分类 × 181 天 = 1,448 行

#### 6.3.4 LEFT JOIN 语义

```sql
LEFT JOIN dim_category cat ON f.category_id = cat.category_id
```

- 使用 `LEFT JOIN` 而非 `INNER JOIN`：保留 `fact_price_clean` 中所有行
- 若 `category_id` 在 `dim_category` 中不存在，`category_name` 为 NULL → 可在应用层检测数据完整性
- 实际上 `fact_price_clean.category_id` 已在清洗阶段从 `dim_product` JOIN 获取，此处的 JOIN 仅用于获取 `category_name`

#### 6.3.5 基期价格指数计算

```
price_index = (当日加权均价 / 基期加权均价) × 100
```

其中基期为 **2026-01-01**（数据集的第一个交易日），基期指数固定为 **100.00**。

**SQL 实现**（子查询获取基期价格）：

```sql
SELECT
    category_id,
    SUM(price * sales_volume) / SUM(sales_volume) AS base_price
FROM fact_price_clean
WHERE price_date = '2026-01-01'
GROUP BY category_id
```

**`any(base.base_price)` 的语义**：基期子查询按 `category_id` 分组后，每个分类只有一行。在外层 `GROUP BY` 中，每个分组内的 `base_price` 值恒定，使用 `any()` 聚合函数取其唯一值。

**示例**：某分类 2026-01-01 加权均价 = 100 元，2026-01-15 加权均价 = 105 元，则：
- 基期日 (01-01)：price_index = 100 / 100 × 100 = **100.00**
- 01-15：price_index = 105 / 100 × 100 = **105.00**

**设计优势**：
- 以相对指数替代绝对价格，消除不同分类价格量级差异（如生鲜 ~20 元 vs 数码 ~2000 元）
- 8 个分类在基期日均为 100.00，可直接在同一 Y 轴上比较涨幅/跌幅
- 符合经济学 CPI 指数的标准表述方式

#### 6.3.6 分区策略

```sql
PARTITION BY toYYYYMM(price_date)
```

- 6 个月 = 6 个分区，约 68 万行/分区
- 查询某月数据时，ClickHouse 自动裁剪无关分区
- 支持按分区 TTL 自动删除过期数据（预留扩展）

### 6.4 SQL 执行计划优化要点

| 优化点 | 说明 |
|--------|------|
| 排序键对齐 | `ORDER BY (category_id, price_date)` 与 `GROUP BY` 键匹配，ClickHouse 可利用排序键加速聚合 |
| 避免运行时 JOIN | `category_id` 在写入 `fact_price_clean` 前已预计算（Python 端 JOIN product 表），聚合时仅需轻量 LEFT JOIN 获取名称 |
| 分区裁剪 | 按 `toYYYYMM(price_date)` 分区，查询某月数据时只扫描一个分区；基期子查询 `WHERE price_date = '2026-01-01'` 也受益于分区裁剪 |
| MergeTree 引擎 | ClickHouse 的向量化执行引擎对 `SUM` 聚合有数量级性能优势 |
| `any()` 函数 | 基期子查询按 `category_id` 分组后每组仅一行，`any(base.base_price)` 避免额外聚合开销 |
| 基期子查询单次求值 | 基期价格在子查询中一次计算出 8 个值（每分类一个），外层聚合时的 JOIN 仅需匹配 8 行，开销可忽略 |

### 7.3 test_ck_sql.py — ClickHouse SQL 逻辑验证（9 项）

| 编号 | 测试名称 | 类型 | 验证内容 |
|------|---------|------|---------|
| TC-CK-01 | `test_clean_price_data_logic` | 单元 | 10 行模拟数据（含 4 条正常、6 条脏数据），验证清洗后保留 4 条 + 脏数据分类统计正确 |
| TC-CK-02 | `test_weighted_avg_price_formula` | 单元 | 手工验证 `Σ(p×v) / Σ(v)`：商品A(10×5)+商品B(20×10) → 250/15 = 16.67，与简单平均 15.0 差异验证 |
| TC-CK-03 | `test_aggregation_correctness` | 单元 | 5 行数据的 GROUP BY (category_id, price_date) 聚合，验证分组数、加权均价、total_sales 精度 |

| TC-CK-05 | `test_ddl_syntax_validation` | 单元 | 关键字检查：`ENGINE = MergeTree()`、`ORDER BY category_id`、`PARTITION BY toYYYYMM(price_date)` |

**设计亮点**：
- TC-CK-06 覆盖 8 个边界条件（None、负数、零、=5000 边界、999999 极端值、正常值 × 3），使用参数化测试思想
